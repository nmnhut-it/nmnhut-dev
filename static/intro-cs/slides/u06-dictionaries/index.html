Dictionaries

A new way to store data: look things up by name, not by position. Then a gentle first look at recursion. Based on MIT 6.0001, Lecture 6.

Lists let us look up by position (index 0, 1, 2). A dictionary lets us look up by a label we choose. That's the big idea today, plus a friendly intro to a function that calls itself.

The Problem: Two Lists That Must Stay in Sync

names  = ["Ana", "John", "Sam"]
grades = ["B",   "A+",   "C"]

# What is John's grade? Find where "John" is, then peek grades:
grades[names.index("John")]   # "A+"  -- clumsy!

Two lists, glued together only by matching positions. Add a name but forget the grade and everything shifts out of line.

Set up the pain: parallel lists force you to juggle a shared index. One slip and name and grade no longer line up. There is a data type built exactly for this.

A Dictionary: Key → Value

grades = {"Ana": "B", "John": "A+", "Sam": "C"}
#          key    value

Curly braces { }. Each entry is a key, a colon, then its value. The key is the label you look things up by.

A dictionary maps keys to values — lookup by name, not by position.

This is the representation thread for the unit. A list is indexed by 0,1,2. A dictionary is indexed by whatever labels you choose. Point out braces vs the square brackets of a list.

Lookup by Name

grades = {"Ana": "B", "John": "A+", "Sam": "C"}

grades["John"]     # "A+"   <- ask by the key
grades["Ana"]      # "B"

Predict grades["Sam"] before you run it. Square brackets hold the key, and out comes its value.

Same square-bracket syntax as a list, but the thing in the brackets is a key, not a position. Have them predict "C" then run. Note there is no searching — the dict jumps straight to the value.

Add and Update

grades = {"Ana": "B", "John": "A+"}

grades["Sam"] = "C"     # NEW key -> adds an entry
grades["Ana"] = "A"     # EXISTING key -> updates it

print(grades)   # {"Ana": "A", "John": "A+", "Sam": "C"}

Same line, two jobs. New key: it's added. Key already there: its value is replaced.

One assignment statement both inserts and overwrites, depending on whether the key already exists. There is only ever one value per key — writing "Ana" again does not make a second Ana.

Gotcha: a Missing Key

grades = {"Ana": "B", "John": "A+"}

grades["Bob"]           # KeyError!  "Bob" was never added

"Bob" in grades         # False  <- safe way to check first
grades.get("Bob")       # None   <- safe: no crash if missing

Asking for a key that isn't there crashes with a KeyError. Use in or .get() to stay safe.

This is the classic beginner surprise. Plain bracket lookup on a missing key raises KeyError — it does not return None. Show the two safe patterns: the in test, and .get which returns None instead of crashing.

Looking Through a Dictionary

grades = {"Ana": "B", "John": "A+", "Sam": "C"}

for name in grades:              # loops over the KEYS
    print(name, "got", grades[name])

grades.keys()      # the keys
grades.values()    # the values

Looping over a dict gives you its keys, one at a time — then use each key to fetch its value.

A for-loop over a dict yields keys, not values — a common confusion. From each key you can reach its value with grades[name]. Mention .keys() and .values() as ways to ask for one side directly.

List or Dictionary?

Use a list

order matters; you want item 1, 2, 3… a plain sequence

Use a dict

you look things up by a label — name→grade, word→count

Storing each student's grade and finding it by name? That's a dictionary — the key is the name, no index juggling.

Give them the decision rule. Lists are for ordered sequences you walk through. Dicts are for "given this label, what's the value?" This is exactly the parallel-lists problem from slide 2, solved.

Example: Count the Words

text = "red blue red red blue"
counts = {}                       # start empty

for word in text.split():         # ["red","blue","red","red","blue"]
    if word in counts:
        counts[word] = counts[word] + 1    # seen before: add 1
    else:
        counts[word] = 1                   # first time: start at 1

print(counts)   # {"red": 3, "blue": 2}

Trace it word by word. The dict is a tally sheet: key = word, value = how many times.

The signature dictionary example. Walk through it: red not in counts, set 1; blue set 1; red already there, becomes 2; and so on. This pattern — build a count keyed by the thing you're counting — comes up everywhere.

A New Idea: a Function That Calls Itself

def countdown(n):
    if n == 0:              # BASE CASE: small enough to answer now
        print("Liftoff!")
    else:
        print(n)
        countdown(n - 1)    # RECURSIVE CASE: call itself on a smaller n

countdown(3)   # 3  2  1  Liftoff!

Every recursion needs a base case (when to stop) and a recursive case that shrinks toward it.

Introduce recursion gently. A function is allowed to call itself. Two must-haves: a base case that stops, and a recursive case on a smaller input. Drop the base case and it never stops — like two mirrors facing each other.

Tiny Example: Factorial

def fact(n):
    if n == 1:                 # base case
        return 1
    return n * fact(n - 1)     # recursive case: n! = n * (n-1)!

fact(4)  # 4 * fact(3)
         # 4 * (3 * fact(2))
         # 4 * (3 * (2 * 1))  = 24

It shrinks 4→3→2→1, hits the base case, then multiplies the answers back up.

Factorial mirrors the math: n! = n times (n-1)!. Trace the unwinding down to fact(1)=1, then the results gather back up to 24. Keep it light — no call-tree diagrams, just the chain.

Quick Check

You run grades["Bob"] but "Bob" was never added. What happens?

Plain bracket lookup on a missing key raises KeyError — it does not return None or add anything. Use "in" to check first, or .get() to get None safely instead of a crash.

Recap & Next

  • A dictionary maps keys to values — look up by name, not position.
  • Add/update with d[key] = value; a missing key raises KeyError.
  • Loop a dict to get its keys; word-count is the classic use.
  • Recursion = a base case to stop + a recursive case that shrinks the input.
