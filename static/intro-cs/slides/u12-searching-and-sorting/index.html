Searching & Sorting

How to find a thing in a list — fast — and why putting the list in order first is the secret. Based on MIT 6.0001, Lecture 12.

Final unit. It pulls the whole course together: a list is a representation, an algorithm is a rule, and how fast it runs is efficiency. Order turns out to be a representation choice that unlocks speed.

The Problem: Find It in a List

Input
a list + a target Question
is it in there? where? Output
its position, or "not found"

Every app does this: a contact, a song, a message. Today: find a number in a list of numbers.

Keep it concrete: search over a list of numbers. Ask students how their phone finds a contact so fast in a list of hundreds. That is the hook for binary search later.

Linear Search — Check One by One

def linear_search(lst, target):
    for i in range(len(lst)):     # look at every spot
        if lst[i] == target:
            return i              # found it here!
    return -1                     # never showed up

Just walk down the list. Works on any list, sorted or not. Worst case (it's not there) you check every item: O(n).

Simple and honest: this is brute force. Returning early on a hit helps the lucky case but not the worst case, which is what Big-O measures. No order needed — that is the key contrast with binary search.

A Smarter Way: The Phone-Book Trick

Looking up "Nguyen" in a paper phone book, you don't start at page 1. You flip to the middle, see if you're too far or not far enough, and throw away half the book. Then repeat.

Each look cuts the problem in half. That only works because the book is in order.

This is binary search in plain language. Stress the precondition out loud: the phone book is alphabetised. If names were in random order, the halving trick would be useless — you couldn't know which half to keep.

Binary Search — Only on a Sorted List

def binary_search(sorted_lst, target):
    lo, hi = 0, len(sorted_lst) - 1
    while lo <= hi:
        mid = (lo + hi) // 2          # the middle spot
        if sorted_lst[mid] == target:
            return mid                # found it
        elif sorted_lst[mid] < target:
            lo = mid + 1              # keep the upper half
        else:
            hi = mid - 1              # keep the lower half
    return -1

Precondition: the list must be sorted. On a jumbled list the go-left / go-right guesses are meaningless.

Peek at the middle, drop the half that can't contain the target. lo and hi mark the piece still in play. Have students trace finding 7 in [1,3,4,7,9,11]. Emphasise: give it an unsorted list and it returns garbage.

Why It's So Fast: O(log n)

n n/2 n/4 … 1

A list of 1000 shrinks 1000 → 500 → 250 → … → 1 in about 10 steps. Halving again and again is what O(log n) means.

Linear search would check up to 1000. Binary search: ~10. That's a huge win.

Concrete numbers land better than a formula: doubling the list adds just one step. 1000 -> ~10, a million -> ~20. That is the payoff of logarithmic growth, first met back in the efficiency units.

Order Is a Representation Choice

Unsorted list

only linear search works → O(n)

Sorted list

binary search unlocked → O(log n)

Same numbers, same data — but choosing to keep them in order lets you search far faster. So: how do we sort?

The unit's representation thread. The values don't change; the arrangement does, and arrangement is a choice you make. Sorting is the price of admission for fast search — which motivates the rest of the deck.

Sorting: Put the List in Order

One easy method — selection sort:

Find

the smallest value still unsorted

Swap

it to the front of the unsorted part

Repeat

with the rest, until done

Like picking players shortest-first and lining them up one at a time.

Selection sort is the most intuitive to trace by hand. Keep the mental model: a growing sorted section on the left, a shrinking unsorted section on the right. Next slide traces it on real numbers.

Trace It: Sort [5, 2, 8, 1]

[5, 2, 8, 1]   # smallest is 1 -> swap to front
[1, 2, 8, 5]   # smallest of rest is 2 -> already there
[1, 2, 8, 5]   # smallest of rest is 5 -> swap with 8
[1, 2, 5, 8]   # last one is already in place -> done!

Do the next list yourself: trace [3, 1, 2] the same way. What's each step?

Walk it live, underlining the sorted prefix as it grows. Four items means three real passes; the last element is forced. Have them trace [3,1,2]: find 1 -> [1,3,2], find 2 -> [1,2,3]. Doing it by hand cements the idea.

How Long Does Sorting Take?

def selection_sort(lst):
    result = list(lst)                      # copy; don't wreck the original
    for i in range(len(result)):            # one pass per position
        smallest = i
        for j in range(i + 1, len(result)): # scan the rest
            if result[j] < result[smallest]:
                smallest = j
        result[i], result[smallest] = result[smallest], result[i]
    return result

A loop inside a loop → roughly n × n steps → O(n²). Slower than searching, but you only do it once.

Two nested loops is the tell for quadratic. n^2 is not great, but the next slide is the punchline: you pay it once and then reap cheap searches forever.

The Big Idea: Sort Once, Search Many

Sort just to search once?

not worth it — the sort alone costs more than one linear search

Search again and again?

sort one time, then every lookup is a cheap O(log n)

A contacts app sorts your names once, then answers thousands of fast searches. The sort pays for itself.

This is the amortisation argument in plain terms. One sort spread across many searches becomes negligible. Real systems keep data sorted (or indexed) exactly so lookups stay fast.

One More Trick: Merge Sort (optional)

For big lists there's a faster sort using divide and conquer:

Divide

split the list in half

Conquer

sort each half

Combine

merge the two sorted halves

This reaches O(n log n) — much better than O(n²) for large lists. (Just a taste; explore it later.)

Optional enrichment only. The takeaway is that clever structure beats brute force again, and that divide-and-conquer keeps reappearing in CS. Don't trace the recursion at this level; plant the seed.

Quick Check

You run binary_search on [5, 2, 9, 1, 7] looking for 9. What's the problem?

Answer: the list is not sorted, so binary search's left/right decisions are invalid — it may return a wrong index or -1. Binary search is only correct on sorted input; that precondition is the whole point of the unit.

Recap & The Whole Course

  • Linear search checks one by one — O(n), works on any list.
  • Binary search halves each time — O(log n), but needs a sorted list.
  • Sorting (like selection sort) puts a list in order so fast search is possible.

The whole course in one line: turn a problem into data (representation), write exact rules over it (algorithms), and care how fast those rules run (efficiency). You can do all three now.

Close the arc. Order is the last representation choice we meet, and it directly buys efficiency. Congratulate them: representation, rules, and efficiency were the three threads all year, and this unit ties them together.
