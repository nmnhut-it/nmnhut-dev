Guess My Number

I'm thinking of a number from 1 to 100. Each guess, I tell you "higher" or "lower." What's your best move?

Guess in order

1, 2, 3, 4... up to 100 tries. Slow.

Guess the middle

50! Every answer throws away half the numbers left.

Play it: I say 73. Guess 50 → "higher" → 75 → "lower"... How many guesses until you're stuck on one number?

The number-guessing game is binary search in disguise. Let a student actually play. The point they should feel: chopping in half is wildly faster than walking up one at a time.

Halving Is Shockingly Fast

Start with 100 numbers. Each guess halves what's left:

100 → 50 → 25 → 13 → 7 → 4 → 2 → 1

Just 7 guesses to pin down any number in 1..100. That "how many times can I halve n" count is called O(log n) — logarithmic time.

Double the range to 200? You need only one more guess. That's what makes log growth amazing.

log base 2 of 100 is about 6.6, so 7 halvings. The killer feature: to handle twice as much input you add a single step, not twice the steps. Do not prove anything — just show the halving chain and the "+1 for double" fact.

Scan Everything vs. Cut in Half

Linear — O(n)

Check items one by one. A list twice as long takes twice as long. Shrinks by one each step.

Logarithmic — O(log n)

Jump to the middle, throw away half. A list twice as long takes just one more step.

Cutting the problem by a factor each step is the whole secret of log time. But halving only works if the list is sorted — that's the price.

Contrast the two signatures: minus one each step (linear) vs cut by a factor each step (log). Sortedness is what lets you know which half to keep. This sets up binary search as the code version of the guessing game.

Binary Search: The Guessing Game in Code

def binary_search(L, target):     # L must be sorted
    low = 0
    high = len(L) - 1
    while low <= high:
        mid = (low + high) // 2       # the "middle guess"
        if L[mid] == target:
            return mid                # found it
        elif L[mid] < target:
            low = mid + 1             # answer is "higher"
        else:
            high = mid - 1           # answer is "lower"
    return -1                         # not in the list

low and high squeeze together, halving the range each loop → O(log n).

Read it as the guessing game: mid is the guess, low/high track "higher/lower". No slicing, so each loop is constant work and the loop runs about log n times. Trace it once on a tiny sorted list live.

Log vs. Linear: See the Gap

n Linear steps (scan) Log steps (halve) 1010~4 1,0001,000~10 1,000,0001,000,000~20

A million items: scanning takes a million checks; halving takes about 20. Same job, wildly different cost.

Let the numbers land. log2 of a million is about 20. This is the payoff slide for log time. Emphasize: the log column barely moves even as n explodes down the rows.

The Slow Direction: Doing the Same Work Twice

def fib(n):
    if n == 0 or n == 1:
        return n
    return fib(n - 1) + fib(n - 2)   # TWO calls each time

To get fib(5) it computes fib(4) and fib(3)... which each split again. The same little answers get recomputed over and over.

Two calls per step makes the work branch and pile up — O(2ⁿ), exponential.

Draw the call tree for fib(5): fib(3) appears twice, fib(2) three times. The cost is the branching, not the answer. This is the opposite of halving — instead of throwing work away, it duplicates it.

Log stays tiny. Linear keeps pace with n. Exponential runs off the page — fib(50) the naive way would take longer than your lifetime.

This is the vivid centerpiece. 2^50 is over a quadrillion. The lesson: the class you land in matters far more than any clever tweak inside a step. Now show how to escape exponential.

The Fix: Remember Your Answers

def fib(n, memo=None):
    if memo is None:
        memo = {}                # a dict: number -> its answer
    if n == 0 or n == 1:
        return n
    if n in memo:                # already worked this out?
        return memo[n]           # reuse it, don't redo it
    memo[n] = fib(n - 1, memo) + fib(n - 2, memo)
    return memo[n]

Store each fib(k) once in a dictionary. Now every value is computed a single time → O(2ⁿ) becomes O(n). This trick is called memoization.

Representation thread: the dict is a key-to-value map (Unit 6) used as a memory of solved subproblems. Same answers, exponentially less work. The branching tree collapses to n distinct values. Note the fresh-dict guard so a new top-level call starts clean.

Quick Check

Guessing a number from 1 to 100 by always picking the middle, about how many guesses do you need in the worst case?

Answer: about 7 — you can halve 100 down to 1 about seven times, so it's O(log n). Scanning in order would be up to 100 (linear). More questions in the L11 quiz.md.

Recap & Next

  • Logarithmic (log n): cut the problem in half each step — barely grows. Binary search, guess-the-number.
  • Linear (n): touch everything once — grows steadily.
  • Exponential (2ⁿ): branching work that redoes itself — explodes. Fix it by remembering answers (memoization).
