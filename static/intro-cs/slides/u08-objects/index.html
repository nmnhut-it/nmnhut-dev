Objects (OOP)

An object bundles data and behavior into one thing. We'll design our own kind of thing — a Dog — and give it its own data and its own actions. Based on MIT 6.0001, Lecture 8.

First taste of object-oriented programming. Keep it concrete: we build ONE small class (Dog) across the whole deck, so every idea lands on something they can picture.

What Is an Object?

Data
what it knows: name, age Behavior
what it can do: have a birthday

An object is one thing that carries its own data and the actions that go with it — packed together.

Big idea for the whole unit: data + behavior in one bundle. A real dog "knows" its name and age and can do things. That pairing is what an object captures.

You've Used Objects All Along

name = "Rex"
name.upper()       # "REX"   <- text has actions built in

pets = ["Rex", "Ada"]
pets.append("Momo")  # a list has actions too

A string holds data (letters) plus actions (.upper()). A list holds data plus actions (.append()). Now you get to invent your own kind of thing.

Ground the abstraction in what they already know. Every value they've met is an object: data plus dot-notation actions. We're about to build one from scratch.

Class vs. Instance

Class

the blueprint — describes what every dog has and can do

Instance

one real dog built from the blueprint

Many from one

one blueprint, as many dogs as you like

class Dog is the blueprint; rex = Dog("Rex", 3) is one dog with its own name and age.

Analogy: a cookie cutter (class) vs the cookies (instances). Writing the class = design the shape once. Making instances = stamp out as many as you want.

Defining a Class

class Dog:
    """A pet dog with a name and an age."""
    # The blueprint starts here.
    # Inside we'll say what data each dog holds
    # and what actions it can do.

The class keyword names a brand-new kind of thing. By convention the name starts with a capital letter: Dog.

Keep it minimal here — an empty-ish blueprint. Capitalized class names are the Python convention. Next slide we fill in the data.

__init__ and self

class Dog:
    def __init__(self, name, age):
        self.name = name    # this dog's own name
        self.age = age      # this dog's own age

__init__ runs automatically when you make a new dog. self means "this particular dog," so self.name is this dog's name.

Two double-underscores each side of init. self is a stand-in for whichever dog is being built. self.name = name stores the value onto that specific object. Don't over-explain — they'll see it work next slide.

Making Real Dogs

rex = Dog("Rex", 3)     # __init__ runs: self.name="Rex", self.age=3
ada = Dog("Ada", 7)     # a second, separate dog

print(rex.name)   # Rex   <- dot reads this dog's data
print(ada.age)    # 7

You pass two values, but __init__ lists three — Python fills self for you with the new dog.

Each call makes a distinct object with its own name and age. rex and ada don't share data. Dot notation (rex.name) reads an attribute, exactly like they've done with library values.

Each Instance Has Its Own Data

rex = Dog("Rex", 3)
ada = Dog("Ada", 7)

rex.age = 4        # only changes Rex
print(rex.age)     # 4
print(ada.age)     # 7   <- Ada is untouched

Changing one dog's data leaves the others alone. Each instance keeps its own copy of name and age.

This kills the common beginner fear that all dogs "share" a variable. The blueprint is shared; the data is per-instance. Good moment to have them predict both prints.

A Method: birthday()

class Dog:
    def __init__(self, name, age):
        self.name = name
        self.age = age

    def birthday(self):
        self.age = self.age + 1    # this dog gets one year older

A method is an action that belongs to the class. It takes self, so it can read and change this dog's data.

A method is just a function that lives inside the class and gets self. birthday changes the dog's own age. Note there's no return — it mutates the object.

Calling a Method

rex = Dog("Rex", 3)
print(rex.age)    # 3

rex.birthday()    # runs birthday with self = rex
print(rex.age)    # 4

rex.birthday() — the dog on the left of the dot becomes self inside the method. That's why it changes Rex's age.

Same dot notation as .append(). No argument passed, but self is filled with rex. Have them trace: age goes 3 then 4. Call it twice and it's 5.

__str__: a Nice Printout

class Dog:
    # ... __init__, birthday as before ...
    def __str__(self):
        return self.name + " (age " + str(self.age) + ")"

print(rex)   # Rex (age 4)
# without __str__ you'd see: <__main__.Dog object at 0x...>

Define __str__ to choose what print shows. It must return a string. Without it, you get an ugly type-and-address default.

__str__ is another double-underscore method Python calls on print. Great for debugging — you see real values, not a memory address. Must return a str, hence str(self.age).

Put It Together

class Dog:
    def __init__(self, name, age):
        self.name = name
        self.age = age
    def birthday(self):
        self.age = self.age + 1
    def __str__(self):
        return self.name + " (age " + str(self.age) + ")"

rex = Dog("Rex", 3)
rex.birthday()
print(rex)        # Rex (age 4)

Trace it: what is rex.age after birthday(), and what does the last line print?

The whole class on one slide: data (init), behavior (birthday), printout (str). This is the shape of every class they'll write. Have them run it and then add a second dog.

Quick Check

A dog is made with d = Dog("Momo", 3), then d.birthday() runs (which does self.age = self.age + 1). What does d.age return now?

Mirrors the L8 quiz Q10 (setter mutates the instance). birthday runs self.age = self.age + 1, so 3 becomes 4. The method changed this dog's own data attribute.

Recap & Next

  • An object bundles data and behavior into one thing.
  • A class is the blueprint; each instance has its own data. __init__ and self build them.
  • Methods add actions; __str__ makes a nice printout.
