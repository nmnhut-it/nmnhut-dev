Classes & Inheritance

Building families of related types: a child class that keeps everything its parent has, then adds or changes a little. Based on MIT 6.0001, Lecture 9.

Unit 8 built a single class (data + behavior). Unit 9 connects classes into families so related types share structure instead of repeating it.

Recap: A Class Bundles Two Things

Data

what the object is — a name, an age

Behavior

what the object can do — methods like speak()

Every object built from a class has the same structure, but its own values.

Quick callback to Unit 8. A class is a blueprint; each instance fills in its own data. Today: what if two classes are almost the same?

Cats and Dogs Are Both Animals

Animal
has a name; can speak Cat
is an Animal + says "Meow" Dog
is an Animal + says "Woof"

A Cat is an Animal. Writing the name code twice would be repeating yourself. Inheritance lets a Cat reuse the Animal parts for free.

Motivate inheritance from real categories. The rule: keep everything the parent has, then add or specialize. Don't-repeat-yourself is the payoff.

The Parent Class: Animal

class Animal:
    def __init__(self, name):
        self.name = name              # store the animal's name

    def get_name(self):
        return self.name

    def speak(self):
        return "Some generic sound"

An ordinary class: it stores a name and knows how to speak.

Keep the base small. self is whichever object we called the method on. speak returns a generic sound for now — children will change it.

A Child Class Inherits Everything

class Cat(Animal):          # Cat's parent is Animal
    pass                    # nothing new yet

tom = Cat("Tom")
print(tom.get_name())       # Tom   <- inherited, for free
print(tom.speak())          # Some generic sound

The name in parentheses — Animal — is the parent. Cat gets its data and methods without repeating them.

class Cat(Animal) means "Cat is an Animal, plus more." Even with an empty body, Cat already has __init__, get_name, and speak from Animal.

Overriding: Change One Method

class Cat(Animal):
    def speak(self):            # OVERRIDE the parent's speak
        return "Meow"

class Dog(Animal):
    def speak(self):
        return "Woof"

Cat("Tom").speak()             # "Meow"
Dog("Rex").speak()             # "Woof"
Cat("Tom").get_name()          # "Tom"  <- still inherited

Redefining a method the parent already has is called overriding.

Each child redefines speak but keeps the inherited get_name. Overriding replaces just that one method; everything else stays inherited.

What a Child Class Can Do

Inherit

get all the parent's data and methods free

Add

give itself brand-new data or methods

Override

redefine a method to change its behavior

Whatever an Animal can do, a Cat can do — plus more.

Three powers of a subclass. This is the whole contract of class Child(Parent): reuse, extend, specialize.

Adding Data With super().__init__

class Cat(Animal):
    def __init__(self, name):
        super().__init__(name)      # run Animal's setup first
        self.lives = 9              # then add Cat's own data

    def speak(self):
        return "Meow"

felix = Cat("Felix")
print(felix.get_name())            # Felix   <- from Animal
print(felix.lives)                 # 9       <- new in Cat

super() means "the parent." super().__init__(name) runs Animal's setup, then Cat adds lives.

super().__init__(name) is the same as Animal.__init__(self, name): it stores self.name. Then Cat layers on lives. Order matters — call the parent first.

Why Call the Parent?

Re-typing self.name = name inside every child class copies the same code again and again.
super().__init__(name) reuses it once. Fix or improve Animal later and every child gets the fix automatically.

This is the payoff of inheritance: don't repeat yourself. One place holds the shared setup.

D.R.Y. in plain words. Shared structure lives in the parent; children trust it. One source of truth is easier to change safely.

Asking "Is It An Animal?"

felix = Cat("Felix")

isinstance(felix, Cat)       # True
isinstance(felix, Animal)    # True   <- a Cat IS an Animal
isinstance(felix, Dog)       # False  <- Felix is not a Dog

isinstance(obj, Class) checks whether an object counts as that type — true for the whole family chain, not just the exact class.

isinstance is the "is-a" test. Because Cat inherits from Animal, felix passes both checks. Handy for writing code that works on any Animal.

Polymorphism: Same Call, Different Result

zoo = [Cat("Tom"), Dog("Rex"), Cat("Felix")]

for animal in zoo:
    print(animal.speak())    # Meow / Woof / Meow

The loop never asks "cat or dog?" Each object runs its own speak. One method name, type-appropriate behavior — that's polymorphism.

Polymorphism = one interface, right behavior per type. Python looks up speak on each object's real class. This lets you write general code over a mixed list.

Quick Check

Both Animal and Cat define speak. Which one runs for Cat("Tom").speak()?

Python looks in the object's own class first and stops at the first match. Cat.speak wins; the parent's version is not run. The deepest override always wins.

Recap & Next

  • class Child(Parent) inherits all the parent's data and methods; it can add new ones or override them.
  • super().__init__(...) reuses the parent's setup — don't repeat yourself.
  • isinstance answers "is-a" for the whole family; polymorphism lets the same call do the right thing per type.

Inheritance is shared structure across related types — the representation idea of this unit.

Bridge onward. The two ideas most worth keeping: super() for shared setup, and polymorphism over a mixed list. Try building Animal -> Cat/Dog yourself.
