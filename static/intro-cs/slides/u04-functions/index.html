Functions

Give a useful chunk of code a name, then reuse it as many times as you like. A function is a rule: data in → value out. Based on MIT 6.0001, Lecture 4.

Units 1-3 wrote code line by line. Today we bundle a rule under a name so we can reuse it. Representation thread: a function names and reuses a rule.

Why Bother With Functions?

Say you check "is this number even?" in three different places. Copy-paste it three times, and every fix means fixing it three times.

The pain

The same code, copied everywhere.

The fix

Write it once, give it a name, reuse it.

The bonus

Fix one place, fixed everywhere.

A good program is measured by what it does, not how many lines it is.

Motivation is reuse and maintainability. This is decomposition, gently: break a job into named pieces. Do not use the word "abstraction" as jargon yet.

A Function Is a Black Box

Data goes in
a number The rule runs
hidden inside A value comes out
the answer

You use a microwave without knowing its wiring: press a button, food gets hot. A function is the same — you feed it input and take the output, without re-reading how it works inside.

The projector analogy from L4, gentled to a microwave. Black box = you trust the input/output, you do not peek inside. This is the representation lens: data in, value out.

Your First Function

def greet():
    print("Hello!")
    print("Welcome to Python.")

greet()      # runs the two lines -> Hello! / Welcome to Python.

def means "a definition follows." The indented lines belong to the function. Nothing happens until you call it by writing its name with ().

Defining is not running. Point out: def line, colon, indented body. The call greet() is what actually runs the body. Have them predict then run.

Give It an Input

def greet(name):        # name is a parameter -- an input slot
    print("Hello,", name)

greet("Mai")            # name becomes "Mai"  -> Hello, Mai
greet("Tan")            # name becomes "Tan"  -> Hello, Tan

The name in the def line is a parameter — an empty slot. The value you pass in the call fills it. Same function, different input, different result.

Parameter = the slot in the def line. Argument = the actual value passed at the call. Keep it intuitive: the slot gets filled at call time. Reuse with different inputs is the whole point.

return: Hand a Value Back

def double(n):
    return n * 2         # hands the value back to whoever called

result = double(5)       # result is 10
print(result + 1)        # 11  -- we can keep computing with it

return ships one value out of the box. The call double(5) is replaced by its answer, 10 — so you can store it, print it, or do more math with it.

Return is the most important line. The call expression becomes the returned value. Contrast with print on the next slide.

return vs print — the None Trap

def double_return(n):
    return n * 2         # hands 10 back

def double_print(n):
    print(n * 2)         # only SHOWS 10, hands nothing back

a = double_return(5)     # a is 10
b = double_print(5)      # prints 10, but b is None !

print shows a value on screen. return gives it back so you can reuse it. Forget return and you get None — Python's word for "no value."

The classic gotcha. If you see None leak out, you printed inside instead of returning. print is for humans to read; return is for the rest of your program to use.

A Real One: is_even

def is_even(number):
    """Input: a whole number. Returns True if it is even."""
    return number % 2 == 0

print(is_even(4))    # True
print(is_even(7))    # False

The text in """triple quotes""" is a docstring — a short note saying what goes in and what comes out. Anyone can now use is_even without reading how it works.

Recall % from Unit 1. The docstring is the label on the box: inputs and output. A user trusts the docstring; they do not re-derive the body. Keep it one gentle sentence.

Names Inside Stay Inside

def add_tax(price):
    tax = price * 0.1      # tax is LOCAL -- lives only inside
    return price + tax

total = add_tax(100)       # 110
print(tax)                 # ERROR -- tax does not exist out here

A name created inside a function is local: it exists only while the function runs, then disappears. That's a feature — the outside world can't get tangled up in a function's private names.

Scope kept intuitive: local names are private and temporary. Do not draw stack frames for 15-year-olds. The takeaway: what happens in the function stays in the function, except the returned value.

Predict, Then Run

def area_of_square(side):
    return side * side

small = area_of_square(3)      # ?
big   = area_of_square(10)     # ?
print(small, big)

Trace it: what does side become each call? What are small and big? Predict first, then run to check.

Two calls, two separate runs, side is 3 then 10. Answer: 9 and 100. This cements: one function, reused with different inputs, each call independent.

Bonus: A Function Can Take a Function

def do_twice(action):     # action is itself a function!
    action()
    action()

def wave():
    print("wave")

do_twice(wave)            # wave / wave

In Python a function is just another value, so you can hand one to another function. You'll use this later — for now, just enjoy that it works.

Optional/light higher-order, exactly one gentle example. No closures, no lambdas for 15-year-olds. Skip if short on time.

Quick Check

What does this print?
def f(x): x = x + 1; return x — then x = 3; z = f(x); print(x)

Inside f, x = x + 1 changes only f's private local x to 4. That local vanishes when f returns, so the outside x is still 3. z is 4, but we printed x. This is the local-names idea in action.

Recap & Next

  • A function names a rule so you can reuse it: data in → value out.
  • def defines it; calling it with () runs it.
  • return hands a value back; print only shows it — forget return and you leak None.
  • Names created inside a function are local and private.
