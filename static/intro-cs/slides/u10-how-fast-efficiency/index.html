How Fast?

Two programs can give the same answer — but one finishes in a blink and the other takes all day. We measure the work by counting steps, not seconds. Based on MIT 6.0001, Lecture 10.

Unit 10 is our first look at efficiency. The representation idea for today: we represent "how much work" as a step count that grows with the input size — not a stopwatch reading that depends on your computer.

Same Answer, Different Speed

Imagine two ways to find a name in a class list of size n:

Recipe A

check every name, one by one

Recipe B

a clever shortcut that skips most of them

Both find the name. The question isn't "is it correct?" — it's "how much work did it do?"

Correctness is not the whole story. Two correct algorithms can differ wildly in work. Ask the class: for a list of 30 names it won't matter — but for 30 million? That's when the choice of recipe decides whether it finishes at all.

Why a Stopwatch Lies

import time
start = time.time()
find_name(class_list, "Mai")   # run the recipe
print(time.time() - start)     # how many seconds?
  • Run it on a fast gaming PC → small number.
  • Run the same code on an old phone → bigger number.
  • Other apps open? The number changes again.

Seconds measure your computer, not your recipe. We need something fairer.

Timing is real and sometimes useful, but it confounds the algorithm with the hardware and the moment. The same program times differently on different machines, so seconds can't tell us which recipe is smarter. We want a machine-independent measure.

Count the Steps, Not the Seconds

Instead of a stopwatch, count the little actions the recipe does: each compare, each add, each store is one step.

Seconds
depends on the computer Steps
same on every computer

A step count is the same whether you run it on a supercomputer or a calculator. That's the fair measure.

This is the representation move: "work" becomes a count of primitive operations. We pretend each step costs the same and just count how many happen. It's an estimate, but a machine-independent one — the whole class can agree on it.

Let's Count One

def total(n):
    result = 0            # 1 step
    for i in range(n):    # the loop body runs n times...
        result = result + i   # ...doing 1 step each time
    return result         # 1 step
# steps = 1 + n + 1  =  n + 2

The loop runs n times, so the count depends on n. Bigger list → more steps.

If n is 100, how many steps? If n is 1000?

Walk it live: setup is 1, the loop body happens n times, the return is 1, giving n + 2. The key realization is that the count is a formula in n — it grows as the input grows. That growth is the thing we really care about.

Class 1 — Constant (always the same)

def first_item(seats):
    return seats[0]   # grab position 0 — always 1 step

Whether the list has 10 items or 10 million, this does the same tiny amount of work. No loop, no growth.

We call this constant time, written O(1) — "order 1".

Constant time is the dream: the work does not depend on n at all. Looking up one position in a list is O(1). The "1" doesn't mean literally one step — it means a fixed number that never grows with the input.

Class 2 — Linear (one loop)

def contains(seats, wanted):
    for person in seats:      # loop over all n people
        if person == wanted:  # 1 compare each time
            return True
    return False

One loop over n things → about n steps. Double the list, double the work.

We call this linear time, written O(n).

A single pass over the input is linear: n items, one unit of work each, so the step count grows in step with n. "Recipe A" from slide 2 lives here. Doubling the input doubles the work — a straight line.

Class 3 — Quadratic (a loop inside a loop)

def any_duplicate(seats):
    for a in seats:           # n times...
        for b in seats:       # ...n times each  ->  n x n
            if a is not b and a == b:
                return True
    return False

For each of n people, you scan all n again → n × n steps.

We call this quadratic time, written O(n^2). This one gets scary fast.

A nested loop multiplies: the outer runs n times, and each outer pass runs the inner n times, giving n times n. This is where growth turns painful — the next slide's table makes it vivid. Nested loops are the classic sign of quadratic work.

Watch It Grow

n    O(1)    O(n)    O(n^2) 10    1    10    100 100    1    100    10,000 1,000    1    1,000    1,000,000 1,000,000    1    1,000,000    1,000,000,000,000

Constant stays flat. Linear keeps up. Quadratic explodes — a trillion steps for a million-item list.

Let the numbers land. At a million items, O(1) is still one step, O(n) is a million (fine), and O(n^2) is a trillion (hopeless in real time). The shape of the growth — not the exact count — is what decides whether a program is usable.

Big-O: Shorthand for "How It Grows"

We had n + 2 steps for the loop. As n gets huge, the + 2 stops mattering — 1,000,002 is basically a million.

n + 2 → O(n)   drop the small + 2 5 × n → O(n)   drop the multiplier n×n + 3×n + 9 → O(n^2)   keep the biggest part

Big-O keeps only the fastest-growing part and throws away constants. It names the shape, not the exact count.

Big-O is a deliberate simplification: for large n only the dominant term matters, so we drop additive and multiplicative constants. n + 2, 3n, and 5n + 100 are all O(n) — same shape. Students should read O(n^2) as "grows like n squared," not a precise step count.

Quick Check

A loop inside a loop, each running n times. How does the number of steps grow?

Answer: O(n^2). The outer loop runs n times and each pass runs the inner loop n times, so n times n. The last option is the trap — step counts don't depend on the machine; that was the whole point of counting steps instead of seconds.

Recap & Next

  • Seconds measure your computer; counting steps measures the recipe fairly.
  • How the step count grows with n gives the class: O(1) constant, O(n) linear, O(n^2) quadratic.
  • One loop is linear; a loop inside a loop is quadratic and explodes.
  • Big-O keeps only the fastest-growing part — the shape, not the exact count.
