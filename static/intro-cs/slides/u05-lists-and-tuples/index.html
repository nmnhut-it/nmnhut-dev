Lists & Tuples

One name for a whole collection of things — data you can index, loop over, and change. Based on MIT 6.0001, Lecture 5.

So far a variable held one value. A list holds many, in order. This unit is about collections — and one sneaky bug (aliasing) that trips up almost everyone the first time.

One Name, Many Values

scores = [88, 72, 95, 60]
print(scores)      # [88, 72, 95, 60]

A list is an ordered collection — many values kept in a row, under one name, in square brackets [ ].

Instead of s1, s2, s3, s4, you keep them all in one place: scores.

The representation move: a bunch of related values is data too. "Ordered" means position matters — first, second, third. A list can hold ints, strings, anything, even mixed.

Reach In by Position

pets = ["cat", "dog", "fish"]
pets[0]        # "cat"   <- counting starts at ZERO
pets[2]        # "fish"
pets[-1]       # "fish"  <- -1 means "last"
len(pets)      # 3       <- how many items

Predict pets[1] before you run it. Each item has an index — its position, counting from 0.

Zero-based indexing is the classic first surprise. Position 0 is the first item, so the last index is len - 1. Negative indexes count from the back. len tells you the size.

Loop Over Every Item

scores = [88, 72, 95]
total = 0
for s in scores:      # s becomes each item in turn
    total = total + s
print(total)          # 255

A for loop walks through the list one item at a time — no counting, no index needed.

This is the payoff of a collection: one loop handles four items or four thousand. Trace it: s is 88, then 72, then 95; total grows each pass. Connect to the loops from Unit 2.

Growing a List: append

guests = ["Ana", "Ben"]
guests.append("Cy")    # adds to the END, in place
print(guests)          # ["Ana", "Ben", "Cy"]
len(guests)            # 3

.append(x) changes the list you already have — it does not make a new one. Lists can grow and shrink; this is called being mutable.

Mutable = changeable in place. append adds one item at the end. Note there is no new list here — same list, now longer. This "changes in place" idea matters a lot in three slides.

Sorting: Two Ways

nums = [3, 1, 2]

nums.sort()            # CHANGES nums -> [1, 2, 3], gives back None
new = sorted(nums)     # RETURNS a new sorted list

# TRAP:
nums = nums.sort()     # now nums is None!  sort() returns nothing

.sort() reorders the list in place. sorted(list) hands you a brand-new sorted list and leaves the original alone.

Number-one beginner bug: writing L = L.sort() and getting None. sort mutates and returns None; sorted returns the new list. Pick sorted when you want to keep the original order too.

Tuples: Locked Lists

point = (4, 7)      # round brackets = a TUPLE
point[0]            # 4     <- indexing works like a list
point[0] = 9        # TypeError -- tuples can't be changed

A tuple is an ordered collection too, but fixed — once made, it can't change. Use it for things that belong together and shouldn't move, like an (x, y) point.

Immutable = cannot be changed. Same indexing and len as a list, but no append, no item assignment. Good for fixed pairs. A one-item tuple needs a comma: (5,), not (5).

Unpacking & Returning Two Values

x, y = (4, 7)       # unpack: x = 4, y = 7
a, b = b, a         # swap two names in ONE line!

def min_max(nums):
    return min(nums), max(nums)   # packs a tuple

low, high = min_max([5, 2, 9])    # low = 2, high = 9

A function returns just one thing — but that one thing can be a tuple, so you get two values back at once.

Unpacking spreads a tuple across names. The swap works because the right side is built first. Returning min, max is the everyday reason tuples exist. Caller unpacks into low, high.

The Empty-List Edge Case

bag = []            # a valid, empty list
len(bag)            # 0
bag[0]              # IndexError -- nothing at position 0!

for item in bag:    # loop body simply never runs
    print(item)

Always ask: what if the list is empty? Indexing it crashes, but looping over it is safe — it just does nothing.

Empty is a real, common case (a search that finds nothing, input not given yet). len is 0, any index is an error, and a for loop runs zero times. Checking for empty is a habit that prevents crashes.

Aliasing: Two Names, One List

a = ["red", "green"]
b = a               # NOT a copy -- b is another name for a's list
b.append("blue")

print(a)   # ["red", "green", "blue"]   <- a changed too!
print(b)   # ["red", "green", "blue"]

b = a does not copy the list. Both names point at the same list, so changing one shows up in the other.

The centerpiece bug. Assignment copies the reference, not the contents — like two nicknames for one person. This surprises everyone once. Draw it: two arrows, one box.

Cloning: Make a Real Copy

a = ["red", "green"]
b = a[:]            # CLONE -- [:] builds a brand-new list
# b = list(a)       # same idea, another way
b.append("blue")

print(a)   # ["red", "green"]           <- untouched!
print(b)   # ["red", "green", "blue"]

Want a separate copy? Clone it with a[:] or list(a). Now they are two different lists, and changing one leaves the other alone.

Compare directly with the previous slide: only the b = a line changed, to b = a[:]. Now there are two boxes, two arrows. This is the fix for the aliasing surprise.

Gotcha: Passing a List to a Function

def add_zero(items):
    items.append(0)     # changes the CALLER's list!

my_list = [1, 2, 3]
add_zero(my_list)
print(my_list)          # [1, 2, 3, 0]  <- surprise

def add_zero_safe(items):
    items = items[:]    # work on a clone
    items.append(0)
    return items

A function's parameter is an alias of the list you passed in. Mutate it and you mutate the original — unless you clone first.

Same aliasing rule, now across a function call. The parameter and the argument share one list. Clone at the top if the function shouldn't disturb the caller's data. Very common real bug.

Quick Check

After warm = ['red', 'yellow', 'orange']; hot = warm; hot.append('pink'), what are warm and hot?

Answer: index 1. hot = warm aliases — one list, changed through hot and seen through warm. The trap is thinking assignment copies the list. To copy, use warm[:].

Recap & Next

  • A list [ ] is an ordered collection: index it, len it, loop over it, append to it.
  • .sort() changes in place (returns None); sorted() gives a new list.
  • A tuple ( ) is fixed — great for pairs and returning two values.
  • b = a shares one list (aliasing); b = a[:] makes a real copy — so does passing a list to a function.
