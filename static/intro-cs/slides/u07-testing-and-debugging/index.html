Testing & Debugging

How to find out if your program is right, and how to hunt down why it is wrong. Based on MIT 6.0001, Lecture 7.

By now students can write real programs with functions, lists, and dictionaries. The new skill this unit: proving your code works, and fixing it when it doesn't.

"It runs" is not "it works"

Your code runs with no red errors. Does that mean it's correct? Not even close.

Testing = compare expected (what you say should happen) with actual (what the program really does).

Representation thread: a test turns "is my code right?" into data — two values you can line up and compare. Same move as always: turn a fuzzy question into something exact.

What a Test Really Is

Pick an input Say the expected answer Run it Compare 
print(double(5))    # I EXPECT 10
# if it prints 10  -> pass
# if it prints 7   -> a bug! start debugging
Decide the expected answer BEFORE you run — otherwise you'll just believe whatever the screen shows. Match = move on; mismatch = you found a bug.

Three Kinds of Test Cases

Normal

the everyday case
average([2, 4, 6])

Boundary

the smallest / extreme case
average([5])

Edge

the weird, empty, or tricky case
average([])

Bugs love the corners. Always ask: what's the emptiest, biggest, or strangest input?

The empty list is the classic edge case: dividing by len([]) is dividing by zero. Testing only "normal" inputs is how bugs slip through to real users.

A Buggy Discount Calculator

def final_price(price, discount):
    # discount is a percent, like 20 for 20% off
    return price - discount          # BUG?

print(final_price(100, 20))   # EXPECT 80 ... actual: 80  ?

It prints 80 — looks right! Now try final_price(50, 20). Expected 40. What does it actually give?

final_price(50, 20) gives 30, not 40. The normal-looking test (100, 20) passed by luck. A second test exposes the real bug: it subtracts the percent as if it were dollars.

Step 1 of Debugging: Read the Error

scores = [90, 85, 70]
print(scores[3])

# Traceback (most recent call last):
#   File "grades.py", line 2, in <module>
#     print(scores[3])
# IndexError: list index out of range

The traceback tells you the file, the line, and the kind of error. Read it bottom-up.

Beginners panic at red text and ignore it. The last line is the actual error type and message; the line above points at the exact code. It's a free clue, not a scolding.

A Few Errors You'll Meet

IndexError — list spot doesn't exist: [1,2,3][5] KeyError — missing dictionary key TypeError — wrong type: "a" + 5 ValueError — right type, bad value: int("twelve") ZeroDivisionError — dividing by zero

The tricky one has no error at all: a logic bug — it runs fine but gives the wrong answer.

Don't memorize a giant list. Recognize these five. Logic bugs (like the discount one) get no message, so testing is the only way to catch them.

Step 2: Shrink It and Print

Smallest example
that still breaks Check the middle
print a value halfway Pick a half
bug is where values go wrong 
total = 0
for s in scores:
    total = total + s
    print("so far:", total)   # peek at the value each step
Bisection: check a value in the middle. Right there? Bug is later. Wrong? Bug is earlier. print() is your flashlight — it shows what the machine actually holds, not what you assume.

try / except: Catch a Crash

try:
    n = int(input("How many people? "))
    print("Slices each:", 8 / n)
except ZeroDivisionError:
    print("Can't split a pizza among 0 people!")
except ValueError:
    print("Please type a whole number.")

Risky lines go in try. If they blow up, Python jumps to except instead of crashing.

A user typing "zero" or "banana" would crash a bare program. try/except turns a scary traceback into a friendly message. List specific errors so each gets its own reply.

assert: A Sanity Check

def average(scores):
    assert len(scores) != 0, "no scores given!"
    return sum(scores) / len(scores)

An assert says "this had better be true." If it's false, the program stops right there with a clear message — before a bad value spreads.

Use try/except for messy outside data; use assert to guard your own assumptions.

assert stops trouble at the door. Without it, average([]) would divide by zero deep inside; with it, you get "no scores given!" pointing right at the cause.

Quick Check

Which error does Python raise for [1, 2, 3][5]?

Answer: IndexError — asking for spot 5 in a 3-item list. KeyError is the dictionary version (a missing key). The list has spots 0, 1, 2 only.

Recap & Next

  • Testing = compare expected vs actual; try normal, boundary, and edge cases.
  • Debug like a detective: read the traceback, shrink the example, print to see values.
  • Use try/except to catch crashes; use assert to guard your assumptions.
